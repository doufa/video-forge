# VideoForge 开发路线图 (Roadmap)

[English](roadmap_en.md)

## 项目愿景

面向知识区 UP 主的"本地优先"视频自动化工作台。核心差异化：**不用 AI 生成画面，而是智能检索和组合真实素材**，保证学科严谨性。

---

## 实施原则

> **先验证核心表现力，再补全智能链路，最后实现产品化**

---

## Phase 1: 渲染闭环验证 ✅ 已完成

**目标**：验证 HyperFrames 能否渲染出"花生视频"风格的成片

| 项目 | 详情 |
|------|------|
| **预估时间** | 1 周 |
| **实际时间** | 2026-05-10 完成 |
| **技术选型** | HyperFrames + GSAP + Tailwind CSS |

**选型理由**：
- HyperFrames：HTML 转视频，支持复杂 CSS 动效，比传统 FFmpeg 合成更灵活
- GSAP：成熟的动画库，生态丰富，性能优秀
- 放弃方案：Remotion（需要 React 生态）、FFmpeg 纯合成（动效能力弱）

**交付物**：
- [x] HTML/CSS 视频模板（动态字幕、梗图飞入、缩放转场）
- [x] HyperFrames 渲染器集成
- [x] 手动数据 → MP4 输出验证

## Phase 2: 竞品 RAG 编剧链路 ✅ 已完成

**目标**：跑通"竞品字幕 → 提取结构 → 生成新脚本"的 RAG 链路

| 项目 | 详情 |
|------|------|
| **预估时间** | 2 周 |
| **实际时间** | 2026-05-15 完成 |
| **技术选型** | DeepSeek-V3 + Edge-TTS + WhisperX |

**选型理由**：
- DeepSeek-V3：中文理解能力强，JSON 输出稳定，硅基流动免费额度
- Edge-TTS：微软免费 TTS，质量尚可，零成本
- WhisperX：开源字级对齐，比原版 Whisper 精度高
- 放弃方案：GPT-4（成本高）、Azure TTS（需付费）、原版 Whisper（无字级对齐）

**交付物**：
- [x] `ScriptWriterSkill` - DeepSeek RAG 编剧
- [x] `TTSSkill` - Edge-TTS 配音生成
- [x] `TimestampAlignSkill` - WhisperX 字级时间戳
- [x] 结构化 JSON 分镜表输出

## Phase 3: 云端素材检索 ✅ 已完成

**目标**：实现云端素材自动获取（绕过本地冷启动问题）

| 项目 | 详情 |
|------|------|
| **预估时间** | 1 周 |
| **实际时间** | 2026-05-20 完成 |
| **技术选型** | yt-dlp + LLM 关键词翻译 |

**选型理由**：
- yt-dlp：支持 YouTube/Bilibili 等 1000+ 站点，活跃维护
- LLM 关键词翻译：中文主题 → 英文关键词，提升 YouTube 召回率
- 放弃方案：Pexels API（素材类型受限）、自建爬虫（维护成本高）

**交付物**：
- [x] `YtdlpProvider` - YouTube/Bilibili 素材下载
- [x] 英文关键词 Prompt 优化
- [x] 降级策略（纯色占位图）

## Phase 4: 端到端组装与渲染 ✅ 已完成

**目标**：一键输出示例科普视频

| 项目 | 详情 |
|------|------|
| **预估时间** | 1 周 |
| **实际时间** | 2026-05-22 完成 |
| **技术选型** | 自研 Pipeline Runner |

**选型理由**：
- 自研调度器：简单直接，避免引入 Airflow/Prefect 等重框架
- YAML 配置驱动：易于调整流程顺序和参数
- 放弃方案：LangGraph（状态机过于复杂）、Airflow（部署成本高）

**交付物**：
- [x] `PipelineRunner` 核心调度器
- [x] 脚本 → 配音 → 素材 → 渲染 全链路串联
- [x] `test_e2e.py` 端到端测试

## Phase 5: 资源库基础设施 ✅ 已完成

**目标**：建立本地素材管理系统

| 项目 | 详情 |
|------|------|
| **预估时间** | 1 周 |
| **实际时间** | 2026-05-25 完成 |
| **技术选型** | SQLite + JSON Sidecar |

**选型理由**：
- SQLite：零配置，单文件，适合本地优先场景
- JSON Sidecar：与素材文件同目录，便于迁移和备份
- 放弃方案：PostgreSQL（部署复杂）、纯文件系统（查询效率低）

**交付物**：
- [x] SQLite 数据库 (`storage/database.py`)
- [x] JSON Sidecar 元数据 (`storage/sidecar.py`)
- [x] 路径工具 (`utils/paths.py`)

## Phase 6-7: CLIP 嵌入与 FAISS 向量检索 ✅ 已完成

**目标**：让本地素材"可被 AI 检索"

| 项目 | 详情 |
|------|------|
| **预估时间** | 2 周 |
| **实际时间** | 2026-05-28 完成 |
| **技术选型** | OpenAI CLIP + FAISS |

**选型理由**：
- CLIP：图文对齐能力强，支持中英文文本查询
- FAISS：Meta 出品，高效向量检索，支持 GPU 加速
- 放弃方案：Pinecone（云服务，隐私顾虑）、Milvus（部署复杂）、Chroma（性能一般）

**交付物**：
- [x] `CLIPEmbedder` - 视频帧向量化
- [x] `VectorStore` - FAISS 索引管理
- [x] `LocalFaissProvider` - 本地优先检索

## Phase 8-9: 自动入库与 RAG 模板 ✅ 已完成

**目标**：自动化素材入库 + 叙事结构模板库

| 项目 | 详情 |
|------|------|
| **预估时间** | 2 周 |
| **实际时间** | 2026-06-02 完成 |
| **技术选型** | PySceneDetect + MD5/pHash + LLM 结构提取 |

**选型理由**：
- PySceneDetect：基于内容的场景切分，比固定间隔切片更智能
- MD5 + pHash：精确去重 + 感知去重，双重保障
- LLM 结构提取：从竞品视频学习叙事模式
- 放弃方案：固定时长切片（浪费存储）、仅 MD5 去重（无法处理转码）

**交付物**：
- [x] `DedupService` - 重复检测
- [x] 自动切片入库流程
- [x] `RAGTemplateExtractor` - 叙事结构提取
- [x] 模板库存储

## Phase 10: Web UI 工作台 🔜 下一阶段

**目标**：提供可视化操作界面

| 项目 | 详情 |
|------|------|
| **预估时间** | 4 周 |
| **技术选型** | React + TypeScript + FastAPI + WebSocket |

**选型理由**：
- React：生态成熟，组件库丰富（计划用 shadcn/ui）
- FastAPI：Python 异步框架，与现有代码无缝集成
- WebSocket：实时推送渲染进度
- 备选方案：Streamlit（快速原型）、Gradio（简单交互）

**规划交付物**：
- [ ] React + TypeScript 前端
- [ ] 项目看板（创建、列表、状态）
- [ ] 脚本可视化编辑器（分镜预览、拖拽排序）
- [ ] 素材库管理面板（搜索、预览、标签）
- [ ] 渲染监控与质量报告

## Phase 12: 三级 QA 系统完善 📋 规划中

**目标**：防范盲盒式产出

| 项目 | 详情 |
|------|------|
| **预估时间** | 2 周 |
| **技术选型** | FFprobe + Whisper + Gemini Flash |

**选型理由**：
- FFprobe：音视频元数据检测，检测黑屏/静音
- Whisper：ASR 对比字幕偏差
- Gemini Flash：多模态一致性检查，成本低

**规划交付物**：
- [ ] 基础关：FFprobe 黑屏/静音检测
- [ ] 同步关：Whisper 字幕偏差校验
- [ ] 语义关：多模态一致性核查（画面 vs 旁白）
